## Common Lifecycle And Host Rules

Source references:

- `apps/web/src/app/shared/directives/shared/base-directive.ts`
- `apps/web/src/app/shared/directives/shared/directive-host.ts`
- `apps/web/src/app/shared/directives/shared/directive-helpers.ts`

Common behavior:

- Browser-only behavior should be guarded by `host.isBrowser`, `IS_BROWSER_PLATFORM`, or `isPlatformBrowser`.
- DOM listeners should be registered through the base DOM listener manager or cleaned up in `ngOnDestroy`.
- Options should be updated through signal input bindings rather than ad hoc mutation.
- Errors should be logged through `LoggerService` or directive `executeSafely`.
- Directives should not implement business authorization. They own DOM behavior only.

Host bindings/listeners:

- `DraggableDirective` has Angular `host` metadata for classes/styles.
- Most other directives attach DOM listeners directly through base helpers, Renderer2, or the third-party library instance.
- D3 directives render directly into host elements and clear SVG/DOM on destroy.

Reusable sample patterns:

- Put each directive in a focused demo panel with state, config, event log, and API buttons.
- Keep config objects in TypeScript and bind them through directive inputs.
- Log lifecycle events for complex directives such as Dropzone, FullCalendar, Tree, Inputmask, and Tagify.
- Use template references for directive APIs only when the source directive exports an API.
- Add clear accessibility labels to jump links, buttons, menus, and interactive examples.

Caveats from sample analysis:

- External selector names and import paths are not Velora APIs.
- The sample is a showcase, not a domain workflow. Do not copy generic directive demo panels into workspaces.
- Inputmask-heavy examples need form-specific validation review before use in standard controls.
- Maxlength display belongs to textarea usage in current Velora form controls.

## Directive Usage Rules

- Prefer component wrappers for common domain UI, but use directives for low-level DOM behavior.
- Keep directive configs in TypeScript objects for complex integrations.
- Always subscribe to error/validation outputs for file, calendar, tree, mask, and tag inputs.
- Do not use directive demos as workspace acceptance proof. Every directive-backed interaction must affect domain state.
- When using `vlVeloraInputmask` in form controls, verify masked/unmasked values, disabled/readonly propagation, and validators.
- `vlVeloraMaxlength` is accepted in standard controls only for textarea. Inputs use native `maxlength` plus form validators.
